00 Lab Overview

This lab demonstrates IAM least privilege through three progressively sophisticated access patterns, one per group. Each group teaches a distinct real-world model for how permissions should flow in production environments.

Least Privilege

Grant only the permissions required. Start from deny-all, add only what is needed, use explicit denies for critical exclusions.

RBAC

Policies live on groups and roles, not individual users. Users inherit access purely through group membership.

Permission Boundaries

A hard ceiling on maximum permissions. Even AdministratorAccess is cut down if the action falls outside the boundary.

STS + Session Policies

Exchange long-lived credentials for short-lived scoped creds. Session policies narrow the role's permissions at assume-time.

Effective permissions formula

Effective = (Identity policy) ∩ (Permission boundary) ∩ (Session policy if STS used). All three layers must allow an action for it to succeed. Any one layer denying it is enough to block it.

P2
Phase 2
Create Roles reordered
v1 bug v1 created roles in Phase 6 but inline policies in Phase 3 already referenced role ARNs. Fixed by creating roles first. ARNs are deterministic so the policies could be written without the roles existing, but creating roles first is clearer and avoids confusion when verifying.

Each role needs two things: a trust policy (who can assume it) and permissions (what they can do after assuming). Trust policies are set at role creation time — permissions are attached after. The developers group uses direct managed policy permissions only, so no role is created for alice.

ops-role — assumed by ops-* users for EC2 operations

bash trust-ops-team.json + ops-ec2-policy.json 
# Trust policy: any user whose ARN matches ops-* in this account
cat > trust-ops-team.json << EOF
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "AWS": "arn:aws:iam::${ACCOUNT_ID}:root" },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringLike": {
        "aws:PrincipalArn": "arn:aws:iam::${ACCOUNT_ID}:user/ops-*"
      }
    }
  }]
}
EOF

aws iam create-role \
  --role-name ops-role \
  --assume-role-policy-document file://trust-ops-team.json \
  --description "EC2 ops role — ops team must assume this to act"

# Inline policy on the role: EC2 start/stop, no delete/terminate
cat > ops-ec2-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EC2OpsAccess",
      "Effect": "Allow",
      "Action": ["ec2:Describe*", "ec2:StartInstances", "ec2:StopInstances"],
      "Resource": "*",
      "Condition": {
        "StringEquals": { "aws:RequestedRegion": "us-east-1" }
      }
    },
    {
      "Sid": "DenyDestructive",
      "Effect": "Deny",
      "Action": ["ec2:TerminateInstances", "ec2:DeleteSecurityGroup"],
      "Resource": "*"
    }
  ]
}
EOF

aws iam put-role-policy \
  --role-name ops-role \
  --policy-name ops-ec2-inline \
  --policy-document file://ops-ec2-policy.json

readonly-role — assumed by readonly-* users with broad ReadOnly permissions

Why ReadOnlyAccess on the role is intentional

The role gets full ReadOnlyAccess deliberately. The per-session narrowing happens when carol calls AssumeRole with a session policy that restricts to S3 only. Same role, different effective permissions per caller. This is the power of session policies.

bash trust-readonly-team.json + attach ReadOnlyAccess 
cat > trust-readonly-team.json << EOF
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "AWS": "arn:aws:iam::${ACCOUNT_ID}:root" },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringLike": {
        "aws:PrincipalArn": "arn:aws:iam::${ACCOUNT_ID}:user/readonly-*"
      }
    }
  }]
}
EOF

aws iam create-role \
  --role-name readonly-role \
  --assume-role-policy-document file://trust-readonly-team.json \
  --description "Full ReadOnly — session policy narrows per-session"

aws iam attach-role-policy \
  --role-name readonly-role \
  --policy-arn arn:aws:iam::aws:policy/ReadOnlyAccess
verify both roles created
aws iam list-roles \
  --query "Roles[?contains('ops-role readonly-role', RoleName)].{Name:RoleName,Created:CreateDate}" \
  --output table

Managed Policy on developers Group

The developers group gets one thing: the dev-s3-limited managed policy attached directly. This gives dev-alice always-on, standing S3 access without any role exchange. No sts:AssumeRole permission is granted — that pattern belongs to ops-team and readonly-team, which are demonstrated in P4.

sed -i cross-platform note

sed -i "..." on Linux works as written. On macOS, it requires sed -i '' "...". This lab uses envsubst to substitute variables into JSON templates instead, which is identical on both platforms.

Why no role for developers?

The developers group demonstrates pure direct RBAC — the managed policy grants standing access without any role exchange. This is deliberately distinct from the ops-team and readonly-team patterns which both require sts:AssumeRole to act at all. Keeping each group's access model separate makes the contrast between patterns explicit and easier to reason about.

P6
Phase 6
Permission Boundary on dev-alice
What a permission boundary does

A boundary is a managed policy used as a ceiling — it defines the maximum permissions the user can ever have. The effective permission for any action is the intersection of the user's identity policy AND the boundary. A boundary does not grant permissions — it only caps them. If the boundary allows S3 but denies IAM, the user can never call any IAM action even if AdministratorAccess is attached later.

bash dev-boundary.json — create, save ARN, apply to alice 
cat > dev-boundary.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowedServices",
      "Effect": "Allow",
      "Action": [
        "s3:*",
        "ec2:Describe*",
        "sts:AssumeRole",
        "cloudwatch:GetMetricData",
        "logs:Describe*",
        "logs:Get*"
      ],
      "Resource": "*"
    },
    {
      "Sid": "DenyPrivilegedServices",
      "Effect": "Deny",
      "Action": [
        "iam:*",
        "organizations:*",
        "account:*",
        "budgets:*"
      ],
      "Resource": "*"
    }
  ]
}
EOF

aws iam create-policy \
  --policy-name dev-permission-boundary \
  --policy-document file://dev-boundary.json \
  --description "Hard ceiling for dev-alice — no IAM, no billing regardless of identity policy"

export BOUNDARY_ARN=$(aws iam list-policies \
  --query "Policies[?PolicyName=='dev-permission-boundary'].Arn" \
  --output text)
echo "BOUNDARY_ARN=$BOUNDARY_ARN"

# Apply boundary to dev-alice only
aws iam put-user-permissions-boundary \
  --user-name dev-alice \
  --permissions-boundary $BOUNDARY_ARN
verify boundary applied
aws iam get-user --user-name dev-alice \
  --query "User.PermissionsBoundary.PermissionsBoundaryArn" \
  --output text
# Expected: arn:aws:iam::ACCOUNT_ID:policy/dev-permission-boundary

Phase 8
Enable CloudTrail timing fix

CloudTrail records every API call as a JSON event. IAM and STS events include the caller identity, action, resource, and any error code. When bob assumes a role using session name bob-ops-session, that session name appears in every CloudTrail event from that session — making it trivially easy to trace role usage back to an individual user.

bash create trail bucket, policy, trail, start logging 
export TRAIL_BUCKET="cloudtrail-iam-lab-${ACCOUNT_ID}"
aws s3 mb s3://$TRAIL_BUCKET --region us-east-1

# CloudTrail requires this exact bucket policy — it will not write without it
cat > trail-bucket-policy.json << EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSCloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::${TRAIL_BUCKET}"
    },
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::${TRAIL_BUCKET}/AWSLogs/${ACCOUNT_ID}/*",
      "Condition": {
        "StringEquals": { "s3:x-amz-acl": "bucket-owner-full-control" }
      }
    }
  ]
}
EOF

aws s3api put-bucket-policy \
  --bucket $TRAIL_BUCKET \
  --policy file://trail-bucket-policy.json

aws cloudtrail create-trail \
  --name iam-lab-trail \
  --s3-bucket-name $TRAIL_BUCKET \
  --is-multi-region-trail

aws cloudtrail start-logging --name iam-lab-trail
confirm trail is active
aws cloudtrail get-trail-status \
  --name iam-lab-trail \
  --query "{Logging:IsLogging,LastDelivery:LatestDeliveryTime}"
# IsLogging should be true

Why budgets:* instead of ce:GetCostAndUsage

v1 used aws ce get-cost-and-usage to test billing denial. The problem: Cost Explorer must be enabled separately in the AWS account under Billing preferences. If it's not enabled, the API returns a service error (not an IAM AccessDenied), which looks like the test failed for the wrong reason. budgets:describe-budgets is part of the same DenyBilling statement and does not need explicit service enablement.

T4 ops-bob: Zero Direct Access → Assume ops-role → EC2 fix

The key demonstration for the ops-team pattern. Bob is denied at every AWS service call until he assumes the role. Then EC2 describe works. Destructive EC2 actions remain denied via the explicit deny in the role's inline policy.

v1 bug v1 used i-fake123 as the test instance ID. EC2 validates instance ID format before calling IAM — i-fake123 fails format validation (InvalidInstanceID.Malformed) before IAM evaluates the policy, so the test returned the wrong error. Fixed: i-1234567890abcdef0 is a valid format. EC2 will return InvalidInstanceID.NotFound (the instance doesn't exist in your account), but that error comes after IAM evaluation — you're testing IAM, not EC2 inventory.
bash bob zero-access proof → assume ops-role → EC2 works, terminate blocked

aws ec2 describe-instances --profile bob --region us-east-1

T5 readonly-carol: AssumeRole + Session Policy Narrowing

Session policy mechanics

readonly-role has AWS-managed ReadOnlyAccess — this allows EC2, RDS, CloudWatch, S3, and every other read action. When carol calls AssumeRole she passes a session policy allowing only s3:Get* and s3:List*. Effective permissions = role policy ∩ session policy. EC2 reads are denied because the session policy doesn't include EC2 — even though the role allows them. A session policy can only restrict, never expand beyond what the role allows.

bash carol-session-policy.json + assume with session policy injected 
# Session policy — written at call-time, narrows what carol gets from the role
cat > carol-session-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:Get*", "s3:List*"],
    "Resource": "*"
  }]
}
EOF

# Assume role AND pass session policy in the same call
read -r AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN <<(
  aws sts assume-role \
    --role-arn "arn:aws:iam::${ACCOUNT_ID}:role/readonly-role" \
    --role-session-name carol-readonly-session \
    --policy file://carol-session-policy.json \
    --duration-seconds 900 \
    --profile carol \
    --query "Credentials.[AccessKeyId,SecretAccessKey,SessionToken]" \
    --output text
)
export AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN

# Confirm identity shows carol's role session
aws sts get-caller-identity
# UserId: assumed-role/readonly-role/carol-readonly-session

# S3 reads work — both role AND session policy allow them
aws s3 ls
aws s3 cp s3://dev-bucket-${ACCOUNT_ID}/test-object.txt -
# Expected: success — session policy allows s3:Get*

# EC2 reads blocked — role allows them but session policy omits EC2
aws ec2 describe-instances --region us-east-1
# Expected: AccessDenied — role allows, session doesn't include EC2 → intersection = deny

# S3 writes blocked — session policy only has Get*/List*, no Put*
echo "carol write attempt" | aws s3 cp - s3://dev-bucket-${ACCOUNT_ID}/carol-write.txt
# Expected: AccessDenied

unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN
ALLOWs3:ListAllMyBuckets → role allows (ReadOnlyAccess), session allows (s3:List*) — intersection = allow
DENYec2:DescribeInstances → role allows (ReadOnlyAccess), session does NOT include EC2 — intersection = deny
DENYs3:PutObject → session policy only lists Get*/List* — Put* not in session = deny

T6 Boundary Blocks Privilege Escalation fix

Simulate a misconfiguration: someone accidentally attaches AdministratorAccess directly to alice. The permission boundary should make this a non-event — alice still cannot call IAM actions.

v1 bug v1 ran only simulation for this test and immediately cleaned up, giving no chance to verify the boundary works with live credentials. This version adds a live call after simulation to confirm the runtime enforcement matches the predicted simulation result.
bash attach AdministratorAccess → simulate → live call → confirm boundary wins 
# Attach AdministratorAccess to alice directly (simulating misconfiguration)
aws iam attach-user-policy \
  --user-name dev-alice \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

# Step 1: simulate — boundary should still deny IAM actions
aws iam simulate-principal-policy \
  --policy-source-arn "arn:aws:iam::${ACCOUNT_ID}:user/dev-alice" \
  --action-names "iam:CreateUser" "iam:DeleteUser" "iam:CreatePolicy" "iam:AttachUserPolicy" \
  --resource-arns "arn:aws:iam::${ACCOUNT_ID}:user/*" \
  --query "EvaluationResults[*].{Action:EvalActionName,Decision:EvalDecision}" \
  --output table
# All four → implicitDeny — AdministratorAccess means nothing outside the boundary

# Step 2: live call — confirm runtime matches simulation
aws iam list-users --profile alice
# Expected: AccessDenied — boundary denies iam:* at runtime too

# S3 still works (boundary allows s3:*)
aws s3 ls --profile alice
# Expected: lists buckets

# Step 3: clean up the misconfiguration
aws iam detach-user-policy \
  --user-name dev-alice \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

T7 IAM Policy Simulator UI — Console Step

Console — one step only

T8 CloudTrail Audit timing fix

v1 bug v1 said "wait ~5 minutes." CloudTrail's lookup-events API shows events within 15 minutes of occurrence for management events. Plan for up to 15 minutes. The S3-based log files take even longer and require manual download to parse.

CloudTrail audit key insight

Notice that when bob ran EC2 commands, the CloudTrail Username field shows bob-ops-session — the session name from the --role-session-name flag. This is how you trace role usage back to a specific individual in an audit. Without meaningful session names, you'd only know the role was assumed — not by whom or why. This is why session naming conventions matter in production.
